Step 1 of 6 · POST
Submit your counter-offer
HR's initial offer is sitting in your inbox. ₹14L. 2 days WFH. No joining bonus. You're worth more than that.

You type your counter-offer and hit send. That's a POST request — you're asking the server (HR Portal) to create a new negotiation record in the store (their system). If successful, you'll get back a 201 Created with your new negotiation ID.
# What you're sending under the hood:
POST /negotiations HTTP/1.1
Host: hr-portal.company.com
Content-Type: application/json
{
  "candidate_name": "...",
  "salary": 1800000,
  "wfh_days": 4,
  "joining_bonus": 100000
}
POST /negotiations — your counter-offer
Response

Step 2 of 6 · GET
Check what HR has on file
You sent your counter-offer. But did they record it correctly? You ask: "What exactly does your system show for my negotiation?"

That's a GET request — read-only, no changes, just fetch. The server looks up your negotiation ID in the store and returns whatever it finds. This is the most common REST call in any system.
# Read-only — no body needed
GET /negotiations/your-id HTTP/1.1
Host: hr-portal.company.com
# No Content-Type, no body — GET never has a body
GET /negotiations/{id} — read current terms
This ID was returned by the server when you submitted your counter-offer.

Step 3 of 6 · PUT
Full renegotiation
HR came back with a counter-counter. You've reconsidered your entire position. You want to send a completely new set of terms — not a tweak, a full replacement.

That's a PUT request. You send every field. The server replaces the entire record. If you leave a field out, it disappears. PUT is destructive by design — it's a full overwrite.
# Must send ALL fields — old record fully replaced
PUT /negotiations/your-id HTTP/1.1
Content-Type: application/json
{ /* complete new record */
  "salary": 1700000,
  "wfh_days": 3,
  "joining_bonus": 75000,
  "role": "Senior Engineer"
}
PUT /negotiations/{id} — replace all terms
All 4 fields required. This replaces the entire record — not a merge.
Response

Step 4 of 6 · PATCH
Just bump the salary
HR agreed on 3 days WFH and the joining bonus. One thing left: you want ₹17.5L, not ₹17L. Everything else is settled — you don't want to resend the whole offer and risk them reopening other terms.

That's a PATCH request — send only what changed. The server merges your patch into the existing record: {**existing, **patch}. One field updated. Everything else untouched.
# Only send what changed — server merges the rest
PATCH /negotiations/your-id HTTP/1.1
Content-Type: application/json
{"salary": 1750000}
# wfh_days, joining_bonus, role — untouched
PATCH /negotiations/{id} — update one field
Leave other fields blank — only salary will change.
Response 
Waiting...
Step 5 of 6 · DELETE
You accepted — close the negotiation
You've got what you wanted. ₹17.5L. 3 days WFH. ₹75K joining bonus. You accept.

The HR system no longer needs your negotiation record — you're no longer a candidate in negotiation. So it gets deleted.

DELETE isn't always a bad thing. Here it's graduation. The server returns 204 No Content — success, nothing left to say. Then watch what happens next.
# No body needed — just the ID in the URL
DELETE /negotiations/your-id HTTP/1.1
Host: hr-portal.company.com
# Response: 204 No Content
# The record is gone. The negotiation is over.
DELETE /negotiations/{id} — accept & close
⚠ This will remove your negotiation record

Step 6 of 6 · POST
Welcome aboard
The candidate row is gone. The negotiation is over. But now HR needs to create a brand new resource — an employee record.

Same verb. POST. Different endpoint. Different store. This is the insight: REST verbs are actions, not things. POST always means "create something new" — whether it's a negotiation, an employee, a 5G session, or an AI message.
# New resource, new endpoint, same verb
POST /employees HTTP/1.1
Content-Type: application/json
{
  "name": "...",
  "role": "Senior Engineer",
  "salary": 1750000,
  "start_date": "2026-07-01"
}
# → 201 Created — employee #XXXX
POST /employees — onboard yourself
